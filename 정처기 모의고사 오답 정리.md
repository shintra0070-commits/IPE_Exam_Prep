### 요구사항 명세 단계 주요 기법

# 비정형 명세기법과 정형 명세기법

## 비교

| 구분 | 비정형 명세기법 (Informal Specification) | 정형 명세기법 (Formal Specification) |
|------|------------------------------------------|---------------------------------------|
| 정의 | 자연어를 이용하여 요구사항을 기술하는 방법 | 수학적 개념과 형식 언어를 이용하여 요구사항을 기술하는 방법 |
| 표현 방법 | 자연어, 표, 그림, 유스케이스 등 | 수학적 모델, 논리식, 형식 언어 |
| 장점 | 이해하기 쉽고 작성이 간단하다. | 명확성이 높고 모호성이 없으며 오류 검증이 가능하다. |
| 단점 | 모호성, 중복, 누락이 발생할 수 있다. | 작성이 어렵고 전문 지식이 필요하며 비용이 많이 든다. |
| 적용 분야 | 일반적인 정보시스템 개발 | 항공, 국방, 의료 등 안전성이 중요한 시스템 |
| 대표 기법 | 자연어 명세, 유스케이스, 사용자 시나리오, 프로토타이핑 | Z, VDM, Petri Net, CSP, B Method |

---

## 비정형 명세기법 (Informal Specification)

### 특징
- 자연어를 사용하여 요구사항을 작성
- 이해하기 쉽고 작성이 간편함
- 사용자와 개발자 간 의사소통이 쉬움
- 모호성(Ambiguity)이 발생할 가능성이 높음

### 대표 기법

| 구분 | 내용 |
|------|------|
| **자연어 (Natural Language)** | 사람이 사용하는 일반적인 언어(한국어, 영어 등)로 요구사항을 작성하는 기법이다. 이해하기 쉽지만 모호성이나 중복이 발생할 수 있다. |
| **유스케이스 (Use Case)** | 사용자(Actor)와 시스템 간의 상호작용을 시나리오 형태로 표현하여 시스템의 기능과 요구사항을 명확하게 기술하는 기법이다. |
| **사용자 시나리오 (User Scenario)** | 실제 사용자의 업무 흐름과 사용 환경을 중심으로 요구사항을 이야기 형식으로 기술하는 기법이다. 사용자 경험(UX)을 고려한 요구사항 분석에 활용된다. |
| **프로토타이핑 (Prototyping)** | 시스템의 화면이나 일부 기능을 미리 구현하여 사용자가 직접 확인하고 요구사항을 검증하거나 수정하는 기법이다. |

---

## 정형 명세기법 (Formal Specification)

### 특징
- 수학적 기호와 논리식을 이용하여 요구사항 작성
- 모호성이 거의 없음
- 요구사항 검증 및 오류 발견이 용이
- 작성이 어렵고 전문 지식이 필요함

### 대표 기법

| 구분 | 내용 |
|------|------|
| **Z** | 집합론(Set Theory)과 술어논리(Predicate Logic)를 기반으로 요구사항을 수학적으로 명세하는 기법이다. 명확성과 일관성이 높다. |
| **VDM (Vienna Development Method)** | 데이터와 연산을 수학적으로 정의하여 시스템을 명세하는 기법이다. 소프트웨어의 설계와 검증에 많이 사용된다. |
| **Petri Net** | 상태(State)와 이벤트(Event)를 그래프로 표현하여 병행 처리, 동기화, 자원 공유 등을 모델링하는 기법이다. |
| **CSP (Communicating Sequential Processes)** | 프로세스 간 통신과 동기화를 수학적으로 표현하는 기법이다. 병렬 시스템 및 분산 시스템 분석에 활용된다. |
| **B Method** | 추상 모델(Abstract Machine)과 단계적 정제(Refinement)를 이용하여 시스템을 개발하고 수학적으로 검증하는 기법이다. 안전성이 중요한 시스템 개발에 사용된다. |

### 디자인 패턴 (Design Pattern)

## 디자인 패턴이란?

소프트웨어 설계 과정에서 자주 발생하는 문제에 대해 **재사용 가능한 해결 방법**을 정리한 설계 방법이다.

- 설계의 재사용성 향상
- 유지보수성 향상
- 개발자 간 의사소통 용이
- 코드의 유연성과 확장성 향상

---

### 디자인 패턴 분류

| 구분 | 목적 | 대표 패턴 |
|------|------|-----------|
| **생성(Creational)** | 객체 생성 방식을 정의 | Singleton, Factory Method, Abstract Factory, Builder, Prototype |
| **구조(Structural)** | 클래스와 객체를 조합하여 구조를 설계 | Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy |
| **행위(Behavioral)** | 객체 간의 상호작용과 책임을 정의 | Chain of Responsibility, Command, Interpreter, Iterator, Mediator, Memento, Observer, State, Strategy, Template Method, Visitor |

---

### 1. 생성(Creational) 패턴

| 패턴 | 내용 |
|------|------|
| **Singleton** | 객체를 하나만 생성하여 어디서든 동일한 객체를 사용하도록 하는 패턴 |
| **Factory Method** | 객체 생성을 서브클래스에서 결정하도록 하여 객체 생성 과정을 캡슐화하는 패턴 |
| **Abstract Factory** | 관련 있는 객체들을 하나의 팩토리에서 생성하는 패턴 |
| **Builder** | 객체 생성 과정을 단계별로 분리하여 복잡한 객체를 생성하는 패턴 |
| **Prototype** | 기존 객체를 복사(Clone)하여 새로운 객체를 생성하는 패턴 |

---

### 2. 구조(Structural) 패턴

| 패턴 | 내용 |
|------|------|
| **Adapter** | 호환되지 않는 인터페이스를 연결하여 사용할 수 있도록 변환하는 패턴 |
| **Bridge** | 구현과 추상을 분리하여 각각 독립적으로 확장할 수 있도록 하는 패턴 |
| **Composite** | 객체들을 트리 구조로 구성하여 부분과 전체를 동일하게 처리하는 패턴 |
| **Decorator** | 객체에 기능을 동적으로 추가하는 패턴 |
| **Facade** | 복잡한 서브시스템을 하나의 간단한 인터페이스로 제공하는 패턴 |
| **Flyweight** | 객체를 공유하여 메모리 사용을 최소화하는 패턴 |
| **Proxy** | 실제 객체 대신 대리 객체를 통해 접근을 제어하는 패턴 |

---

### 3. 행위(Behavioral) 패턴

| 패턴 | 내용 |
|------|------|
| **Chain of Responsibility** | 요청을 처리할 수 있는 객체에게 순차적으로 전달하는 패턴 |
| **Command** | 요청을 객체로 캡슐화하여 실행, 취소 등을 지원하는 패턴 |
| **Interpreter** | 문법 규칙을 클래스 형태로 구현하여 언어를 해석하는 패턴 |
| **Iterator** | 컬렉션 내부 구조를 노출하지 않고 순차적으로 접근하는 패턴 |
| **Mediator** | 객체 간의 복잡한 통신을 중재 객체가 담당하는 패턴 |
| **Memento** | 객체의 이전 상태를 저장하고 복원하는 패턴 |
| **Observer** | 객체 상태가 변경되면 관련 객체들에게 자동으로 통지하는 패턴 |
| **State** | 객체의 상태 변화에 따라 행위가 달라지는 패턴 |
| **Strategy** | 알고리즘을 캡슐화하여 필요에 따라 교체하는 패턴 |
| **Template Method** | 알고리즘의 전체 구조는 유지하고 일부 단계만 서브클래스에서 구현하는 패턴 |
| **Visitor** | 객체 구조는 변경하지 않고 새로운 기능을 추가하는 패턴 |

## 모의고사 02회

### 검증(Verification)과 확인(Validation)

### 개념

| 구분 | 검증 (Verification) | 확인 (Validation) |
|------|---------------------|-------------------|
| 정의 | 소프트웨어가 **명세서(요구사항, 설계서)**대로 개발되었는지 확인하는 과정 | 개발된 소프트웨어가 **사용자의 요구사항을 만족하는지** 확인하는 과정 |
| 목적 | 제품을 **올바르게 만들고 있는가?** | **올바른 제품을 만들었는가?** |
| 질문 | *Are we building the product right?* | *Are we building the right product?* |
| 수행 시기 | 개발 과정 중 | 개발 완료 후 또는 테스트 단계 |
| 대상 | 요구사항, 설계서, 코드 | 실행 가능한 소프트웨어 |

---

### 검증(Verification)

### 특징
- 개발 산출물을 검토하는 과정
- 실행하지 않고도 수행 가능(정적 테스트)
- 오류를 조기에 발견할 수 있음

### 대표 기법

| 구분 | 내용 |
|------|------|
| **관리 리뷰(Management Review)** | 프로젝트 진행 상황과 계획을 검토하여 관리적인 문제를 확인하는 기법 |
| **기술 리뷰(Technical Review)** | 전문가가 기술적인 오류를 검토하는 기법 |
| **인스펙션(Inspection)** | 가장 체계적인 리뷰 기법으로, 사전에 준비된 절차와 체크리스트를 이용하여 결함을 찾는 기법 |
| **워크스루(Walkthrough)** | 작성자가 문서를 설명하면서 참석자들과 함께 검토하는 기법 |
| **감사(Audit)** | 독립적인 조직이 표준이나 규정을 준수했는지 확인하는 기법 |

---

### 확인(Validation)

## 특징
- 실제 프로그램을 실행하여 확인
- 사용자의 요구사항 충족 여부를 검증
- 동적 테스트(Dynamic Testing)에 해당

### 대표 기법

| 구분 | 내용 |
|------|------|
| **단위 테스트(Unit Test)** | 개별 모듈이나 함수가 정상적으로 동작하는지 확인하는 테스트 |
| **통합 테스트(Integration Test)** | 여러 모듈을 결합하여 인터페이스와 데이터 흐름을 테스트하는 기법 |
| **시스템 테스트(System Test)** | 전체 시스템이 요구사항에 맞게 동작하는지 테스트하는 기법 |
| **인수 테스트(Acceptance Test)** | 사용자가 시스템을 검토하여 요구사항 충족 여부를 확인하는 테스트 |

---

### 리뷰(Review) 기법

| 기법 | 내용 |
|------|------|
| **관리 리뷰** | 프로젝트 일정, 비용, 계획 등을 검토 |
| **기술 리뷰** | 기술적인 문제를 전문가가 검토 |
| **워크스루** | 작성자가 설명하며 함께 검토 |
| **인스펙션** | 체크리스트를 이용한 공식적인 검토 |
| **감사(Audit)** | 독립 기관이 표준 준수 여부를 검토 |

---

### 테스트(Test) 종류

### 단위 테스트 (Unit Test)
- 프로그램의 가장 작은 단위(모듈, 함수)를 테스트
- 개발자가 수행하는 경우가 많음

### 통합 테스트 (Integration Test)
- 여러 모듈을 결합하여 테스트
- 인터페이스 오류를 발견

### 통합 테스트 방식

| 방식 | 내용 |
|------|------|
| **빅뱅(Big Bang)** | 모든 모듈을 한 번에 통합 |
| **점진적 통합(Incremental)** | 모듈을 하나씩 통합하며 테스트 |

#### 점진적 통합

- **하향식(Top-Down)** → 스텁(Stub) 사용
- **상향식(Bottom-Up)** → 드라이버(Driver) 사용

### EAI

# EAI (Enterprise Application Integration)

## EAI란?

기업 내 **서로 다른 애플리케이션(Application)**을 연계·통합하여 데이터와 업무를 실시간으로 공유할 수 있도록 하는 통합 솔루션이다.

- 기존 시스템을 변경하지 않고 연계 가능
- 시스템 간 데이터 공유 및 업무 통합
- 중복 개발 및 유지보수 비용 감소

---

# EAI 특징

- 서로 다른 시스템을 통합
- 실시간 데이터 연계 가능
- 업무 프로세스 자동화
- 시스템 간 상호 운용성 향상
- 기존 시스템을 유지하면서 통합 가능

---

# EAI 구성 요소

| 구분 | 내용 |
|------|------|
| **Adapter** | 서로 다른 애플리케이션을 EAI와 연결하는 인터페이스 |
| **Broker** | 데이터 전달 및 메시지 중계 역할을 수행 |
| **Messaging** | 시스템 간 메시지를 송수신하는 기능 |
| **Business Process** | 업무 프로세스를 정의하고 관리하는 기능 |

---

# EAI 구축 유형

| 구분 | 내용 |
|------|------|
| **Point-to-Point** | 시스템 간 1:1로 직접 연결하는 방식이다. 구현은 쉽지만 시스템이 늘어날수록 관리가 어렵다. |
| **Hub & Spoke** | 중앙 허브(Hub)를 통해 모든 시스템을 연결하는 방식이다. 관리가 쉽고 유지보수가 용이하다. |
| **Message Bus** | 공통 메시지 버스를 통해 시스템을 연결하는 방식이다. 확장성이 뛰어나고 대규모 환경에 적합하다. |
| **Hybrid** | 여러 구축 방식을 혼합하여 사용하는 방식이다. |

---

# EAI 장점

- 시스템 간 데이터 공유
- 업무 프로세스 통합
- 중복 개발 감소
- 유지보수 비용 절감
- 업무 처리 속도 향상
- 기존 시스템 재사용 가능

---

# EAI 단점

- 구축 비용이 높음
- 초기 설계가 복잡함
- 장애 발생 시 영향 범위가 큼
- 중앙 허브 장애 시 전체 시스템에 영향을 줄 수 있음(Hub & Spoke 방식)

---

# ESB와 비교

| 구분 | EAI | ESB |
|------|-----|-----|

### 스크럼 기법

## 스크럼이란?

애자일(Agile) 개발 방법론의 대표적인 프로젝트 관리 기법으로, 짧은 개발 주기(스프린트)를 반복하면서 고객의 요구사항을 지속적으로 반영하는 개발 방식이다.

---

# 스크럼 특징

- 반복적(Iterative)이고 점진적인(Incremental) 개발
- 짧은 개발 주기인 스프린트(Sprint) 수행
- 고객 요구사항의 변화에 빠르게 대응
- 팀원 간 협업과 의사소통 중시
- 매일 진행 상황을 공유하는 데일리 스크럼 수행

---

# 스크럼 구성 요소

| 구분 | 내용 |
|------|------|
| **제품 백로그(Product Backlog)** | 프로젝트에서 구현해야 할 모든 요구사항 목록 |
| **스프린트 백로그(Sprint Backlog)** | 하나의 스프린트 동안 수행할 작업 목록 |
| **증분(Increment)** | 스프린트 종료 후 완성된 실행 가능한 결과물 |

| **스크럼 마스터(Scrum Master)** | 스크럼 프로세스가 원활하게 진행되도록 지원하고 장애 요소를 제거하는 사람 |

| **스프린트 리뷰(Sprint Review)** | 스프린트 결과물을 고객과 함께 검토하는 회의 |
| **스프린트 회고(Sprint Retrospective)** | 스프린트 종료 후 개선 사항을 논의하는 회의 |

| **ESM (Enterprise Security Management)** | 여러 보안 장비에서 발생하는 로그와 이벤트를 통합 관리하고 모니터링하는 시스템이다. |
| **SIEM (Security Information and Event Management)** | 다양한 보안 장비의 로그를 수집·분석하여 보안 이벤트를 실시간으로 관리하는 시스템이다. |
| **DLP (Data Loss Prevention)** | 중요 정보의 외부 유출을 탐지하고 차단하는 보안 솔루션이다. |
| **CDS (Cross Domain Solution)** | 서로 다른 보안 등급의 네트워크 간 안전한 데이터 전송을 지원하는 보안 솔루션이다. |
| **DRM (Digital Rights Management)** | 디지털 콘텐츠의 저작권 보호 및 사용 권한을 관리하는 기술이다. |
| **허니팟(Honeypot)** | 공격자를 유인하여 공격 기법을 분석하고 침입을 탐지하기 위한 가상의 시스템이다. |
| **이상금융거래탐지시스템(FDS, Fraud Detection System)** | 금융 거래를 실시간으로 분석하여 이상 거래를 탐지하고 금융 사고를 예방하는 시스템이다. |

# 병행 제어 기법 (Concurrency Control)

## 병행 제어란?

여러 트랜잭션이 동시에 데이터베이스를 접근할 때 **데이터의 일관성과 무결성을 유지하기 위해 트랜잭션의 실행 순서를 제어하는 기법**이다.

---

# 병행 제어의 목적

- 데이터의 일관성 유지
- 데이터의 무결성 보장
- 데이터의 신뢰성 확보
- 동시 수행으로 인한 충돌 방지

---

# 병행 제어 기법

| 구분 | 내용 |
|------|------|
| **로킹(Locking)** | 데이터에 Lock을 설정하여 동시에 접근하지 못하도록 제어하는 기법이다. 공유 락(Shared Lock)과 배타 락(Exclusive Lock)을 사용한다. |
| **2단계 로킹 프로토콜(2PL, Two-Phase Locking)** | 락을 획득하는 확장 단계(Growing Phase)와 락을 해제하는 축소 단계(Shrinking Phase)로 구분하여 직렬 가능성을 보장하는 기법이다. |
| **타임스탬프 순서(Time Stamp Ordering)** | 각 트랜잭션에 고유한 시간 정보를 부여하여 시간 순서대로 트랜잭션을 수행하는 기법이다. |
| **낙관적 검증(Optimistic Validation)** | 충돌이 거의 발생하지 않는다고 가정하고 트랜잭션 종료 시점에 충돌 여부를 검사하는 기법이다. |
| **다중 버전 동시성 제어(MVCC, Multi Version Concurrency Control)** | 여러 버전의 데이터를 유지하여 읽기와 쓰기 작업의 충돌을 줄이는 기법이다. |

---

# 로킹(Locking)

| 구분 | 내용 |
|------|------|
| **공유 락(Shared Lock, S-Lock)** | 여러 트랜잭션이 동시에 읽기(Read)는 가능하지만 쓰기(Write)는 불가능하다. |
| **배타 락(Exclusive Lock, X-Lock)** | 하나의 트랜잭션만 읽기와 쓰기가 가능하며 다른 트랜잭션은 접근할 수 없다. |

---

# 병행 수행 시 발생할 수 있는 문제

| 구분 | 내용 |
|------|------|
| **갱신 분실(Lost Update)** | 두 트랜잭션이 같은 데이터를 수정하여 먼저 수행한 결과가 사라지는 현상 |
| **모순성(Inconsistency)** | 하나의 트랜잭션 수행 중 다른 트랜잭션이 데이터를 변경하여 일관성이 깨지는 현상 |
| **연쇄 복귀(Cascading Rollback)** | 실패한 트랜잭션의 결과를 읽은 다른 트랜잭션도 함께 롤백되는 현상 |
| **비완료 의존성(Dirty Read)** | 커밋되지 않은 데이터를 다른 트랜잭션이 읽는 현상 |

---

### lot 관련 용어

| 구분 | 내용 |
|------|------|
| **IoT (Internet of Things, 사물인터넷)** | 사물에 센서와 통신 기능을 탑재하여 인터넷을 통해 사람, 사물, 시스템 간 정보를 주고받는 기술이다. 스마트홈, 스마트팩토리, 스마트시티 등 다양한 분야에서 활용된다. |
| **MQTT (Message Queuing Telemetry Transport)** | IoT 환경을 위한 **경량 메시지 전송 프로토콜**이다. **TCP 기반**이며 **Publish / Subscribe(Pub/Sub)** 방식을 사용한다. Broker를 통해 메시지를 전달하며 저전력·저대역폭 환경에 적합하다. |
| **CoAP (Constrained Application Protocol)** | 자원이 제한된 IoT 장치를 위한 **경량 웹 전송 프로토콜**이다. **UDP 기반**이며 **Request / Response(REST)** 방식을 사용한다. HTTP보다 오버헤드가 작아 IoT 기기 제어와 센서 통신에 적합하다. |

### 공유 매체에 대한 다중 접근 방식

## 다중 접근 방식이란?

여러 사용자가 **하나의 전송 매체(공유 매체)**를 효율적으로 사용하기 위해 접근을 제어하는 방식이다.

---

# 다중 접근 방식

| 구분 | 내용 |
|------|------|
| **CSMA/CD (Carrier Sense Multiple Access with Collision Detection)** | 전송 전에 회선 사용 여부를 확인(Carrier Sense)하고, 충돌(Collision)이 발생하면 이를 감지(Detection)하여 재전송하는 방식이다. 주로 **유선 이더넷(Ethernet)**에서 사용된다. |
| **CSMA/CA (Carrier Sense Multiple Access with Collision Avoidance)** | 충돌을 미리 방지(Avoidance)하기 위해 전송 전에 일정 시간 대기하거나 RTS/CTS를 사용하여 충돌 가능성을 줄이는 방식이다. 주로 **무선 LAN(Wi-Fi)**에서 사용된다. |
| **TDMA (Time Division Multiple Access)** | 하나의 채널을 시간을 기준으로 나누어 각 사용자에게 일정한 시간 슬롯(Time Slot)을 할당하는 방식이다. |
| **FDMA (Frequency Division Multiple Access)** | 하나의 채널을 여러 개의 주파수 대역으로 분할하여 각 사용자에게 서로 다른 주파수를 할당하는 방식이다. |
| **CDMA (Code Division Multiple Access)** | 동일한 주파수와 시간 대역을 사용하지만 사용자마다 서로 다른 고유 코드(Code)를 부여하여 동시에 통신하는 방식이다. |

# 다중 접근 방식 비교

| 구분 | 기준 | 특징 |
|------|------|------|
| **CSMA/CD** | 충돌 감지 | 유선 LAN(Ethernet), 충돌 발생 후 재전송 |
| **CSMA/CA** | 충돌 회피 | 무선 LAN(Wi-Fi), 충돌 발생 전 회피 |
| **TDMA** | 시간(Time) | 시간 슬롯을 나누어 사용 |
| **FDMA** | 주파수(Frequency) | 주파수 대역을 나누어 사용 |
| **CDMA** | 코드(Code) | 고유 코드를 이용하여 동시에 통신 |

### 라우팅 알고리즘

| 내용 |
| --- |
| 목적지까지의 **최적 경로를 산출하기 위한 알고리즘** |

| 구분 | 내용 |
|------|------|
| **거리 벡터 알고리즘 (Distance Vector)** | 인접한 라우터와 **거리 정보(Hop Count)**를 교환하여 최적 경로를 계산하는 방식이다. 벨만-포드(Bellman-Ford) 알고리즘을 사용하며 **RIP**에서 사용된다. |
| **링크 상태 알고리즘 (Link State)** | 네트워크의 전체 토폴로지 정보를 수집하여 최단 경로를 계산하는 방식이다. 다익스트라(Dijkstra) 알고리즘을 사용하며 **OSPF**에서 사용된다. |
| **경로 벡터 알고리즘 (Path Vector)** | 목적지까지의 **경로(Path) 정보**를 이용하여 최적 경로를 선택하는 방식이다. **BGP**에서 사용된다. |
| **하이브리드 알고리즘 (Hybrid)** | 거리 벡터와 링크 상태 알고리즘의 장점을 결합한 방식이다. **EIGRP**에서 사용된다. |

### 절차형 SQL (Procedural SQL)

| 내용 |
|------|
| 일반 SQL에 **절차적 프로그래밍 요소(변수, 조건문, 반복문 등)**를 추가하여 프로그램처럼 작성할 수 있는 SQL이다. |

| 구분 | 내용 |
|------|------|
| **프로시저 (Procedure)** | 특정 작업을 수행하는 SQL의 집합으로, 호출(Call)을 통해 실행되며 반환값은 선택 사항이다. |
| **함수 (Function)** | 특정 연산을 수행하고 **반드시 하나의 값을 반환(Return)**하는 객체이다. SQL문에서 호출하여 사용할 수 있다. |
| **트리거 (Trigger)** | INSERT, UPDATE, DELETE 등의 이벤트가 발생할 때 **자동으로 실행**되는 객체이다. |
| **패키지 (Package)** | 관련된 프로시저, 함수, 변수, 커서 등을 하나의 객체로 묶어 관리하는 객체이다. |

- 조건문(IF, CASE) 사용 가능
- 반복문(LOOP, WHILE, FOR) 사용 가능
- 예외 처리(Exception Handling) 가능
- 모듈화를 통해 재사용성 향상

### 객체지향 설계 원칙 (SOLID)

| 종류 | 설명 |
|------|------|
| **SRP (Single Responsibility Principle, 단일 책임 원칙)** | 하나의 클래스는 하나의 책임만 가져야 하며, 변경의 이유도 하나여야 한다. |
| **OCP (Open-Closed Principle, 개방-폐쇄 원칙)** | 확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 한다. |
| **LSP (Liskov Substitution Principle, 리스코프 치환 원칙)** | 자식 클래스는 부모 클래스를 대체하여 사용해도 동일한 결과를 보장해야 한다. |
| **ISP (Interface Segregation Principle, 인터페이스 분리 원칙)** | 하나의 범용 인터페이스보다 여러 개의 작은 인터페이스로 분리하는 것이 좋다. |
| **DIP (Dependency Inversion Principle, 의존성 역전 원칙)** | 구체적인 구현이 아닌 추상화(인터페이스, 추상 클래스)에 의존해야 한다. |

### 보안 공격 기법

| 공격 유형 | 종류 | 설명 |
|-----------|------|------|
| **취약점 공격** | **제로데이 공격 (Zero-Day Attack)** | 보안 취약점이 공개되거나 패치되기 전에 해당 취약점을 이용하여 공격하는 방식이다. |
| **지능형 지속 공격** | **APT 공격 (Advanced Persistent Threat)** | 특정 대상을 장기간 지속적으로 공격하여 정보를 탈취하는 지능형 지속 공격이다. |
| **사회공학 공격** | **스미싱 (Smishing)** | 문자메시지(SMS)를 이용하여 악성 앱 설치나 개인정보를 탈취하는 공격이다. |
| **사회공학 공격** | **피싱 (Phishing)** | 이메일이나 가짜 웹사이트 등을 이용하여 개인정보를 탈취하는 사회공학적 공격이다. |
| **사회공학 공격** | **파밍 (Pharming)** | DNS를 조작하여 사용자를 가짜 웹사이트로 유도해 개인정보를 탈취하는 공격이다. |
| **사회공학 공격** | **큐싱 (Qshing, QR Phishing)** | QR코드를 이용하여 악성 사이트로 유도하거나 악성 앱 설치를 유도하는 공격이다. |
| **공급망 공격** | **공급망 공격 (Supply Chain Attack)** | 신뢰하는 소프트웨어나 업데이트 서버, 협력업체를 공격하여 최종 대상 시스템을 감염시키는 공격이다. |
| **부채널 공격** | **부채널 공격 (Side Channel Attack)** | 암호 알고리즘 자체가 아닌 전력 소비, 실행 시간, 전자파 등의 물리적 정보를 분석하여 비밀 정보를 탈취하는 공격이다. |
| **악성코드** | **랜섬웨어 (Ransomware)** | 시스템이나 파일을 암호화한 후 복구 대가로 금전을 요구하는 악성코드이다. |
| **악성코드** | **웜 (Worm)** | 자기 복제 기능을 이용하여 네트워크를 통해 스스로 전파되는 악성코드이다. |
| **악성코드** | **트로이 목마 (Trojan Horse)** | 정상 프로그램으로 위장하여 설치된 후 악성 행위를 수행하는 악성코드이다. |
| **악성코드** | **백도어 (Backdoor)** | 정상적인 인증 절차를 우회하여 시스템에 접근할 수 있도록 만든 비밀 통로이다. |
| **악성코드** | **키로거 (Keylogger)** | 사용자의 키보드 입력을 기록하여 계정 정보나 비밀번호를 탈취하는 악성코드이다. |
| **악성코드** | **루트킷 (Rootkit)** | 시스템에 침투한 후 자신의 존재를 숨기며 관리자 권한을 획득하거나 유지하는 악성코드이다. |
| **악성코드** | **악성봇 (Bot)** | 감염된 PC를 공격자가 원격으로 제어하기 위한 악성코드이다. |
| **악성코드** | **봇넷 (Botnet)** | 악성코드에 감염된 여러 컴퓨터를 원격으로 제어하는 네트워크이다. |
| **악성코드** | **멀웨어 (Malware)** | 바이러스, 웜, 트로이 목마, 랜섬웨어 등 시스템에 피해를 주거나 악의적인 행위를 수행하는 모든 악성 소프트웨어를 통칭하는 용어이다. |
| **서비스 거부 공격** | **서비스 거부 공격 (DoS)** | 특정 시스템에 과도한 요청을 보내 정상적인 서비스를 불가능하게 만드는 공격이다. |
| **서비스 거부 공격** | **분산 서비스 거부 공격 (DDoS)** | 여러 대의 감염된 컴퓨터를 이용하여 동시에 공격하는 서비스 거부 공격이다. |
| **서비스 거부 공격** | **SYN Flooding** | TCP의 3-Way Handshake 과정에서 SYN 패킷을 대량으로 전송하여 서버의 연결 대기 큐를 가득 채워 정상적인 연결을 방해하는 공격이다. |
| **서비스 거부 공격** | **Smurf Attack** | ICMP Echo Request를 브로드캐스트 주소로 전송하고 출발지 IP를 피해자의 주소로 위조하여 대량의 응답을 발생시키는 공격이다. |
| **서비스 거부 공격** | **Ping of Death** | 정상 크기보다 큰 ICMP 패킷을 여러 조각으로 분할하여 전송한 후 재조합 과정에서 시스템 오류나 서비스 장애를 유발하는 공격이다. |
| **서비스 거부 공격** | **TearDrop** | 조작된 Fragment Offset 값을 가진 IP 패킷을 전송하여 패킷 재조립 과정에서 시스템 오류를 유발하는 공격이다. |
| **서비스 거부 공격** | **Land Attack** | 출발지 IP와 목적지 IP를 동일하게 조작한 패킷을 전송하여 시스템에 혼란을 일으키는 공격이다. |
| **서비스 거부 공격** | **HTTP GET 플러딩 (HTTP GET Flooding)** | 대량의 HTTP GET 요청을 반복적으로 전송하여 웹 서버의 CPU, 메모리, 네트워크 자원을 고갈시켜 정상적인 서비스를 불가능하게 만드는 애플리케이션 계층 DDoS 공격|
| **서비스 거부 공격** | **HTTP POST 플러딩 (HTTP POST Flooding)** | 대량의 HTTP POST 요청을 전송하여 서버의 처리 자원을 소모시키고 정상적인 서비스를 불가능하게 만드는 애플리케이션 계층(L7) DDoS 공격이다. |
| **웹 공격** | **SQL Injection** | 입력값에 악성 SQL문을 삽입하여 데이터베이스를 조작하거나 정보를 탈취하는 공격이다. |
| **웹 공격** | **XSS (Cross Site Scripting)** | 웹페이지에 악성 스크립트를 삽입하여 사용자의 정보를 탈취하거나 악성 행위를 수행하는 공격이다. |
| **웹 공격** | **CSRF (Cross Site Request Forgery)** | 사용자가 의도하지 않은 요청을 전송하도록 유도하는 웹 공격이다. |
| **세션 공격** | **세션 하이재킹 (Session Hijacking)** | 사용자의 세션 정보를 가로채 정상 사용자로 위장하는 공격이다. |
| **네트워크 공격** | **스니핑 (Sniffing)** | 네트워크를 통해 전송되는 데이터를 가로채는 공격이다. |
| **네트워크 공격** | **스푸핑 (Spoofing)** | IP, MAC, DNS, 이메일 등을 위조하여 정상 사용자나 시스템인 것처럼 속이는 공격이다. |
| **네트워크 공격** | **리플레이 공격 (Replay Attack)** | 정상적인 인증 정보를 가로채 재전송하여 인증을 우회하는 공격이다. |
| **인증 공격** | **무차별 대입 공격 (Brute Force Attack)** | 가능한 모든 비밀번호를 반복적으로 시도하여 계정을 탈취하는 공격이다. |
| **인증 공격** | **사전 공격 (Dictionary Attack)** | 사전에 있는 단어나 자주 사용하는 비밀번호를 이용하여 인증을 시도하는 공격이다. |
| **인증 공격** | **레인보우 테이블 공격 (Rainbow Table Attack)** | 미리 계산된 해시값 테이블을 이용하여 비밀번호를 복원하는 공격이다. |
| **메모리 공격** | **포맷 스트링 공격 (Format String Attack)** | `printf()`와 같은 포맷 함수에 사용자 입력을 검증하지 않고 사용하여 메모리를 읽거나 수정하는 공격이다. |
| **동시성 공격** | **레이스 컨디션 공격 (Race Condition Attack)** | 여러 프로세스나 스레드가 공유 자원에 동시에 접근하는 순간을 악용하는 공격이다. |
| **공격 분석 모델** | **사이버 킬체인 (Cyber Kill Chain)** | 공격자가 목표 시스템을 공격하는 과정을 단계별로 분석하여 공격을 탐지하고 차단하기 위한 보안 모델이다. |

### 비용 산정 모형

| 종류 | 설명 |
|------|------|
| **LOC (Lines of Code) 모형** | 개발할 프로그램의 예상 소스 코드 라인 수(LOC)를 기준으로 개발 비용을 산정하는 모형이다. |
| **맨먼스 (Man-Month) 모형** | 한 사람이 한 달 동안 수행할 수 있는 작업량을 기준으로 개발 기간과 비용을 산정하는 모형이다. |
| **COCOMO (Constructive Cost Model)** | 보헴(Barry W. Boehm)이 제안한 모형으로, 프로그램 규모(LOC)를 기준으로 개발 비용과 기간을 예측하는 모형이다. |
| **Putnam 모형** | 소프트웨어 생명주기 전반의 인력 분포를 레일리(Rayleigh) 곡선으로 가정하여 개발 비용과 인력을 산정하는 모형이다. |
| **기능점수(FP, Function Point) 모형** | 소스 코드의 양이 아닌 사용자에게 제공되는 기능의 수와 복잡도를 기준으로 개발 비용을 산정하는 모형이다. |

### COCOMO 종류

| 종류 | 설명 |
|------|------|
| **Organic(조직형)** | 소규모 프로젝트로 개발 경험이 풍부한 팀이 수행하는 프로젝트에 적합하다. |
| **Semi-Detached(반분리형)** | 중간 규모 프로젝트로 다양한 경험을 가진 개발팀이 수행하는 프로젝트에 적합하다. |
| **Embedded(내장형)** | 하드웨어, 운영체제 등 제약이 많은 대규모 프로젝트에 적합하다. |

### 지역성 (Locality)

| 종류 | 설명 |
|------|------|
| **시간 지역성 (Temporal Locality)** | 한 번 참조한 데이터나 명령어는 가까운 미래에 다시 참조될 가능성이 높은 특성이다. |
| **공간 지역성 (Spatial Locality)** | 한 번 참조한 데이터의 주변 주소에 있는 데이터도 가까운 미래에 참조될 가능성이 높은 특성이다. |
| **순차 지역성 (Sequential Locality)** | 데이터나 명령어가 저장된 순서대로 연속적으로 참조되는 특성이다. |

### 네트워크 서비스 공격 기법

| 종류 | 설명 |
|------|------|
| **DoS (Denial of Service)** | 하나의 시스템에서 대량의 트래픽을 발생시켜 정상적인 서비스 제공을 방해하는 공격이다. |
| **DDoS (Distributed Denial of Service)** | 여러 대의 감염된 시스템(봇넷)을 이용하여 동시에 대량의 트래픽을 발생시켜 서비스를 마비시키는 공격이다. |
| **Ping of Death** | 정상 크기보다 큰 ICMP 패킷을 여러 개로 분할하여 전송한 후 재조립 과정에서 시스템 오류를 유발하는 공격이다. |
| **Smurfing** | 출발지 IP를 공격 대상의 주소로 위조한 ICMP Echo Request를 브로드캐스트하여 대량의 응답을 발생시키는 공격이다. |
| **SYN Flooding** | TCP 3-Way Handshake 과정에서 SYN 패킷만 반복적으로 전송하여 서버의 연결 자원을 고갈시키는 공격이다. |
| **TearDrop** | IP 패킷의 Fragment Offset 값을 조작하여 재조립 과정에서 시스템 오류를 발생시키는 공격이다. |
| **Land Attack** | 출발지 IP와 목적지 IP를 동일하게 설정한 패킷을 전송하여 시스템에 과부하를 발생시키는 공격이다. |
| **UDP Flooding** | 대량의 UDP 패킷을 전송하여 시스템 자원을 고갈시키는 공격이다. |
| **HTTP GET Flooding** | 정상적인 HTTP GET 요청을 대량으로 발생시켜 웹 서버의 자원을 소모시키는 공격이다. |
| **Slowloris** | HTTP 연결을 오랫동안 유지하여 웹 서버의 연결 자원을 모두 소모시키는 공격이다. |

### 결합도 (Coupling)

| 종류 | 설명 |
|------|------|
| **내용 결합도 (Content Coupling)** | 한 모듈이 다른 모듈의 내부 데이터나 내부 로직을 직접 참조하거나 수정하는 경우이다. **결합도가 가장 높다.** |
| **공통 결합도 (Common Coupling)** | 여러 모듈이 전역 변수(Global Data)를 공동으로 사용하는 경우이다. |
| **외부 결합도 (External Coupling)** | 외부 장치, 통신 프로토콜, 파일 형식 등 외부 환경을 여러 모듈이 공유하는 경우이다. |
| **제어 결합도 (Control Coupling)** | 한 모듈이 다른 모듈의 처리 흐름을 제어하는 제어 정보를 전달하는 경우이다. |
| **스탬프 결합도 (Stamp Coupling)** | 필요한 일부 데이터만 사용하면서도 전체 자료구조(레코드, 객체 등)를 전달하는 경우이다. |
| **자료 결합도 (Data Coupling)** | 필요한 데이터만 매개변수(Parameter)로 전달하는 경우이다. **결합도가 가장 낮다.** |

### 응집도 (Cohesion)

| 종류 | 설명 |
|------|------|
| **우연적 응집도 (Coincidental Cohesion)** | 서로 관련 없는 기능들이 하나의 모듈에 모여 있는 경우이다. **응집도가 가장 낮다.** |
| **논리적 응집도 (Logical Cohesion)** | 논리적으로 유사한 기능을 하나의 모듈로 구성하고 제어 플래그에 따라 기능을 수행하는 경우이다. |
| **시간적 응집도 (Temporal Cohesion)** | 특정 시점에 함께 수행되는 기능들을 하나의 모듈로 구성한 경우이다. |
| **절차적 응집도 (Procedural Cohesion)** | 수행 순서가 관련된 기능들을 하나의 모듈로 구성한 경우이다. |
| **통신적 응집도 (Communicational Cohesion)** | 동일한 입력 데이터나 출력 데이터를 사용하는 기능들을 하나의 모듈로 구성한 경우이다. |
| **순차적 응집도 (Sequential Cohesion)** | 한 기능의 출력이 다음 기능의 입력으로 사용되는 경우이다. |
| **기능적 응집도 (Functional Cohesion)** | 하나의 모듈이 하나의 기능만 수행하는 경우이다. **응집도가 가장 높다.** |

#### 응집도 (낮음 → 높음)

**우논시절통순기**

### 회선 제어 (Line Control)

| 설명 |
|------|
| 데이터 통신 시 송신 측과 수신 측 간의 **통신 회선을 설정하고 관리하는 기법**이다. |

| 종류 | 설명 |
|------|------|
| **ENQ / ACK** | 송신 측이 ENQ(Enquiry)를 전송하여 통신 가능 여부를 확인하고, 수신 측이 ACK(Acknowledge)로 응답하는 방식이다. |
| **Polling / Selecting** | 주국(Primary Station)이 종국(Secondary Station)의 송수신을 제어하는 방식이다. Polling은 종국의 송신 여부를 확인하고, Selecting은 데이터를 수신할 종국을 선택한다. |

---

### 흐름 제어 (Flow Control)

| 설명 |
|------|
| 송신 측과 수신 측의 **처리 속도 차이로 인해 데이터 손실이 발생하지 않도록 데이터 전송량을 조절하는 기법**이다. |

| 종류 | 설명 |
|------|------|
| **Stop-and-Wait** | 한 개의 프레임을 전송한 후 ACK를 수신하면 다음 프레임을 전송하는 방식이다. |
| **Sliding Window** | 여러 개의 프레임을 연속으로 전송하고, 윈도우(Window) 크기만큼 ACK를 받아 전송하는 방식으로 전송 효율을 높인다. |

---

### 오류 제어 (Error Control)

| 설명 |
|------|
| 데이터 전송 과정에서 발생하는 **오류를 검출하거나 수정하여 데이터의 신뢰성을 보장하는 기법**이다. |

| 종류 | 설명 |
|------|------|
| **FEC (Forward Error Correction)** | 수신 측에서 오류를 스스로 수정하는 방식으로 재전송이 필요 없다. |
| **Stop-and-Wait ARQ** | 프레임 하나를 전송한 후 ACK를 받으면 다음 프레임을 전송하는 ARQ 방식이다. |
| **Go-Back-N ARQ** | 오류가 발생한 프레임부터 이후의 모든 프레임을 재전송하는 ARQ 방식이다. |
| **Selective Repeat ARQ** | 오류가 발생한 프레임만 선택적으로 재전송하는 ARQ 방식이다. |

### 입력 데이터 검증 및 표현 취약점

| 설명 |
|------|
| 사용자 입력값에 대한 검증이 미흡하거나 출력 시 적절한 처리를 하지 않아 발생하는 보안 취약점이다. |

| 종류 | 설명 |
|------|------|
| **SQL Injection** | 입력값에 악성 SQL문을 삽입하여 데이터베이스를 조회, 수정, 삭제하는 공격이다. |
| **XSS (Cross Site Scripting)** | 웹페이지에 악성 스크립트를 삽입하여 사용자의 브라우저에서 실행되도록 하는 공격이다. |
| **CSRF (Cross Site Request Forgery)** | 사용자가 의도하지 않은 요청을 서버에 전송하도록 유도하는 공격이다. |
| **운영체제 명령어 삽입 (OS Command Injection)** | 입력값을 이용하여 운영체제 명령어를 실행시키는 취약점이다. |
| **경로 조작 및 자원 삽입 (Path Traversal)** | 상위 디렉터리 접근(`../`) 등을 이용하여 허용되지 않은 파일이나 자원에 접근하는 취약점이다. |
| **파일 업로드 취약점 (Unrestricted File Upload)** | 악성 파일을 업로드하여 서버에서 실행되도록 하는 취약점이다. |
| **파일 다운로드 취약점** | 다운로드 대상 파일에 대한 검증이 부족하여 중요 파일이 노출되는 취약점이다. |
| **LDAP Injection** | LDAP 질의문에 악성 코드를 삽입하여 디렉터리 정보를 조회하거나 변조하는 공격이다. |
| **XPath Injection** | XPath 질의문을 조작하여 XML 데이터에 비정상적으로 접근하는 공격이다. |
| **SSI Injection (Server Side Include Injection)** | SSI 명령어를 삽입하여 서버에서 악성 명령을 실행하는 공격이다. |
| **HTTP 응답 분할 (HTTP Response Splitting)** | HTTP 헤더에 개행 문자를 삽입하여 응답을 분리하고 악성 페이지를 생성하는 공격이다. |
| **정수 오버플로우 (Integer Overflow)** | 정수형 변수의 표현 범위를 초과하여 예상치 못한 동작을 유발하는 취약점이다. |
| **보안 기능 결정에 사용되는 부적절한 입력값** | 인증, 권한 등 보안 기능의 결정에 검증되지 않은 입력값을 사용하여 발생하는 취약점이다. |

### 디지털 포렌식 (Digital Forensics)

| 설명 |
|------|
| 컴퓨터, 스마트폰, 서버, 네트워크 등 디지털 기기에서 범죄와 관련된 데이터를 **수집, 보존, 분석, 제출**하여 법적 증거로 활용하는 기술이다. |

| 종류 | 설명 |
|------|------|
| **정당성** | 모든 증거의 수집과 분석은 적법한 절차와 법적 권한에 따라 수행되어야 한다. |
| **재현성** | 동일한 절차를 수행하면 누구나 같은 결과를 얻을 수 있어야 한다. |
| **무결성** | 수집된 증거는 분석 과정에서도 변경·훼손되지 않아야 한다. |
| **신속성** | 디지털 증거는 쉽게 변경·삭제될 수 있으므로 신속하게 수집하고 분석해야 한다. |
| **연계 보관성 (Chain of Custody)** | 증거의 수집, 이동, 보관, 분석 과정을 모두 기록하여 증거의 신뢰성을 유지해야 한다. |

### 회복(Recovery) 기법

| 설명 |
|------|
| 트랜잭션 수행 중 장애가 발생했을 때 데이터베이스를 **일관성 있는 상태로 복구**하는 기법이다. |

| 종류 | 설명 |
|------|------|
| **로그 기반 회복(Log-Based Recovery)** | 트랜잭션 수행 내용을 로그(Log)에 기록해 두었다가 장애 발생 시 로그를 이용하여 복구하는 기법이다. |
| **체크포인트(Checkpoint Recovery)** | 일정 시점마다 데이터베이스와 로그를 저장하여 장애 발생 시 해당 시점부터 복구하는 기법이다. |
| **즉시 갱신(Immediate Update)** | 트랜잭션 수행 중 데이터베이스에 즉시 반영하며, 장애 발생 시 REDO와 UNDO를 수행하는 기법이다. |
| **지연 갱신(Deferred Update)** | 트랜잭션이 완료(Commit)된 후에만 데이터베이스에 반영하며, 장애 발생 시 REDO만 수행하는 기법이다. |
| **그림자 페이징(Shadow Paging)** | 데이터베이스의 원본은 유지하고 변경 내용을 별도의 그림자 페이지에 저장하여 장애 발생 시 원본으로 복구하는 기법이다. |

### 해킹에 사용되는 공격용 도구

| 설명 |
|------|
| 시스템의 취약점을 탐지하거나 불법적으로 접근, 정보 탈취, 서비스 방해 등을 위해 사용하는 도구이다. |

| 종류 | 설명 |
|------|------|
| **스니퍼(Sniffer)** | 네트워크를 통해 전송되는 패킷을 가로채어 분석하는 도구이다. |
| **크래커(Cracker)** | 암호화된 비밀번호나 인증 정보를 해독(크랙)하는 도구이다. |
| **키로거(Keylogger)** | 사용자의 키보드 입력을 기록하여 계정 정보나 비밀번호를 탈취하는 도구이다. |
| **루트킷(Rootkit)** | 시스템에 침투한 후 자신의 존재를 숨기고 관리자 권한을 유지하도록 하는 도구이다. |
| **백도어(Backdoor)** | 정상적인 인증 절차를 우회하여 시스템에 접근할 수 있도록 만든 비밀 통로이다. |
| **봇(Bot)** | 악성코드에 감염되어 공격자의 명령을 수행하는 프로그램이다. |
| **봇넷(Botnet)** | 여러 대의 감염된 컴퓨터(봇)를 원격에서 제어하는 네트워크이다. |
| **트로이 목마(Trojan Horse)** | 정상 프로그램으로 위장하여 설치된 후 악성 행위를 수행하는 프로그램이다. |
| **웜(Worm)** | 자기 복제 기능을 이용하여 네트워크를 통해 스스로 전파되는 악성코드이다. |
| **랜섬웨어(Ransomware)** | 파일이나 시스템을 암호화한 후 복구 대가로 금전을 요구하는 악성코드이다. |

### 소프트 웨어 테스트 원리

| 설명 |
|------|
| 소프트웨어의 결함을 효과적으로 발견하고 품질을 향상시키기 위해 따라야 하는 기본 원칙이다. |

| 종류 | 설명 |
|------|------|
| **결함 집중(Defect Clustering)** | 소수의 모듈에 대부분의 결함이 집중되는 경향이 있다. |
| **살충제 패러독스(Pesticide Paradox)** | 동일한 테스트를 반복하면 새로운 결함을 발견하기 어려우므로 테스트 케이스를 지속적으로 개선해야 한다. |
| **테스팅은 결함이 존재함을 증명(Testing Shows Presence of Defects)** | 테스트는 결함이 없음을 증명하는 것이 아니라 결함이 존재함을 확인하는 과정이다. |
| **완벽한 테스팅은 불가능(Exhaustive Testing is Impossible)** | 모든 입력값과 조건을 테스트하는 것은 현실적으로 불가능하다. |
| **정황에 의존(Context Dependent)** | 테스트 방법은 소프트웨어의 목적과 환경에 따라 달라져야 한다. |
| **오류-부재의 궤변(Absence-of-Errors Fallacy)** | 결함이 없더라도 사용자 요구사항을 만족하지 못하면 소프트웨어는 성공한 것이 아니다. |
| **테스팅의 조기 수행(Early Testing)** | 테스트는 개발 초기부터 수행할수록 결함을 조기에 발견하여 비용을 절감할 수 있다. |

### XML 기반 기술

| 설명 |
|------|
| 사용자가 직접 태그를 정의하여 데이터를 구조적으로 표현하고 저장·전송할 수 있는 **확장 가능한 마크업 언어**이다. |

### 기술 용어 정리
| 구분 | 설명 |
|------|------|
| **DTD (Document Type Definition)** | XML 문서에서 사용할 요소(Element), 속성(Attribute), 문서 구조를 정의하는 문서 형식이다. |
| **XML Schema (XSD)** | XML 문서의 구조와 데이터 타입을 정의하는 언어로, DTD보다 다양한 데이터 타입과 네임스페이스를 지원한다. |
| **XSL (eXtensible Stylesheet Language)** | XML 문서의 출력 형식과 스타일을 정의하는 스타일시트 언어이다. |
| **XSLT (XSL Transformations)** | XML 문서를 HTML, XML, 텍스트 등 다른 형식으로 변환하는 언어이다. |
| **XPath (XML Path Language)** | XML 문서에서 특정 요소나 속성을 경로(Path)로 탐색하는 언어이다. |
| **XQuery** | XML 문서에서 원하는 데이터를 검색하고 추출하기 위한 질의(Query) 언어이다. |
| **DOM (Document Object Model)** | XML 문서를 트리 구조로 메모리에 저장하여 자유롭게 접근하고 수정하는 방식이다. |
| **SAX (Simple API for XML)** | XML 문서를 순차적으로 읽으며 이벤트 기반으로 처리하는 방식으로 메모리 사용량이 적다. |
| **SOAP (Simple Object Access Protocol)** | XML 기반 메시지를 이용하여 웹 서비스 간 데이터를 교환하는 통신 프로토콜이다. |
| **WSDL (Web Services Description Language)** | 웹 서비스의 기능, 메시지 형식, 접근 방법 등을 XML 형식으로 정의하는 언어이다. |

| **시퀀스 다이어그램(Sequence Diagram)** | 객체 간 메시지의 시간 순서에 따른 상호작용을 표현한다. |
| **커뮤니케이션 다이어그램(Communication Diagram)** | 객체 간의 메시지 교환과 연결 관계를 중심으로 상호작용을 표현한다. |

| **타이밍 다이어그램(Timing Diagram)** | 시간의 흐름에 따른 객체의 상태 변화를 표현한다. |
| **상호작용 개요 다이어그램(Interaction Overview Diagram)** | 여러 상호작용 다이어그램을 활동 흐름 형태로 표현한다. |

### RAID란 ?

| 설명 |
|------|
|여러개의 하드 디스크에 일부 중복된 테이터를 나눠서 저장하는 기술이고 디스크 어레이라고도 한다.|

# RAID 0 ~ RAID 6

| RAID | 설명 | 장점 | 단점 | 최소 디스크 수 | 장애 허용 |
|------|------|------|------|----------------|-----------|
| **RAID 0 (Striping)** | 데이터를 여러 디스크에 분산 저장하여 성능을 높이는 방식 | 읽기/쓰기 속도가 매우 빠름, 저장공간 100% 활용 | 디스크 1개만 고장 나도 전체 데이터 손실 | 2개 | ❌ 불가능 |
| **RAID 1 (Mirroring)** | 동일한 데이터를 여러 디스크에 복제하여 저장하는 방식 | 데이터 복제, 높은 안정성, 복구가 쉬움 | 저장공간이 절반만 사용됨 | 2개 | ⭕ 1개(쌍마다) |
| **RAID 2** | 비트 단위 분산 저장과 해밍 코드(ECC)를 사용하는 방식 | 오류 검출 및 복구 가능 | 구현이 복잡하고 비용이 높아 거의 사용하지 않음 | 여러 개 | ⭕ 가능 |
| **RAID 3** | 바이트 단위 분산 저장과 전용 패리티 디스크를 사용하는 방식 | 대용량 순차 데이터 처리에 유리 | 패리티 디스크 병목 발생 | 3개 | ⭕ 1개 |
| **RAID 4** | 블록 단위 분산 저장과 전용 패리티 디스크를 사용하는 방식 | RAID 3보다 랜덤 읽기 성능 향상 | 패리티 디스크 병목 발생 | 3개 | ⭕ 1개 |
| **RAID 5** | 블록 단위 분산 저장과 분산 패리티를 사용하는 방식 | 성능과 안정성의 균형, 가장 많이 사용 | 디스크 장애 시 복구 시간이 오래 걸림 | 3개 | ⭕ 1개 |
| **RAID 6** | RAID 5에 이중 분산 패리티를 추가한 방식 | 디스크 2개까지 장애 허용, 높은 안정성 | 쓰기 성능이 RAID 5보다 낮음 | 4개 | ⭕ 2개 |

### 특정 타깃을 노리는 공격 기법

| 공격 기법 | 설명 | 특징 |
|-----------|------|------|
| **스피어 피싱 (Spear Phishing)** | 특정 개인이나 조직을 대상으로 신뢰할 수 있는 사람인 것처럼 위장하여 이메일이나 메시지를 보내 정보를 탈취하는 공격 | 특정 대상 공격, 맞춤형 피싱 |
| **APT (Advanced Persistent Threat)** | 특정 조직을 장기간 지속적으로 공격하여 내부에 침투한 후 정보를 수집하거나 탈취하는 공격 | 장기 침투, 지속적인 정보 수집 |
| **워터링 홀 (Watering Hole)** | 공격 대상이 자주 방문하는 웹사이트를 해킹하여 악성코드를 배포하는 공격 | 자주 방문하는 사이트 이용 |
| **스미싱 (Smishing)** | 문자메시지(SMS)를 이용해 악성 앱 설치나 개인정보 입력을 유도하는 공격 | SMS 기반 피싱 |
| **비싱 (Vishing)** | 전화를 이용하여 금융정보나 개인정보를 탈취하는 공격 | 음성(Voice) 기반 피싱 |
| **웨일링 (Whaling)** | CEO, 임원 등 고위 경영진을 대상으로 수행하는 피싱 공격 | 고위 관리자 대상 |
| **파밍 (Pharming)** | DNS 정보를 변조하여 정상 사이트 대신 가짜 사이트로 접속시키는 공격 | 주소는 정상처럼 보임 |
| **BEC (Business Email Compromise)** | 기업 이메일을 사칭하여 송금이나 기밀정보 제공을 유도하는 공격 | 기업 이메일 사칭 |
| **크리덴셜 스터핑 (Credential Stuffing)** | 유출된 계정과 비밀번호를 여러 사이트에 자동으로 입력하여 로그인하는 공격 | 동일 비밀번호 재사용 악용 |
| **사회공학 공격 (Social Engineering)** | 사람의 심리를 이용하여 비밀번호나 중요 정보를 알아내는 공격 | 기술보다 사람을 노림 |
| **이블 트윈 공격(Evil Twin Attack)** | 정상적인 무선 AP(Wi-Fi)와 동일한 이름(SSID)의 가짜 AP를 설치하여 사용자의 접속을 유도한 뒤 개인정보나 통신 내용을 탈취하는 공격 | 가짜 Wi-Fi(AP) 생성, 중간자 공격(MITM) 가능 |

| **형상 관리 도구 (Configuration Management Tool)** | 소스코드와 변경 이력을 관리하여 협업을 지원하는 도구 | Git, SVN, CVS |
| **커버리지 분석 도구 (Coverage Tool)** | 테스트 수행 시 코드가 얼마나 실행되었는지 측정하는 도구 | JaCoCo, Cobertura, Emma |
| **UI 테스트 자동화 도구** | 웹이나 모바일 화면의 동작을 자동으로 테스트하는 도구 | Selenium, Appium, Cypress |

### 객체 지향 구성 요소

| 구성요소 | 설명 | 예시 |
|----------|------|------|
| **객체(Object)** | 데이터(속성)와 기능(메서드)을 함께 가지는 독립적인 실체 | 자동차 객체, 학생 객체 |

### 데이터베이스 암호화 기법

| 암호화 기법 | 설명 | 특징 |
|-------------|------|------|
| **API 방식** | 애플리케이션에서 DB에 저장하거나 조회하기 전에 데이터를 암호화·복호화하는 방식 | 애플리케이션 수정 필요, DBMS와 독립적 |
| **Plug-in 방식** | DB 서버에 암호화 모듈(Plug-in)을 설치하여 데이터를 암호화하는 방식 | 애플리케이션 수정이 적음, 구축이 비교적 쉬움 |
| **TDE (Transparent Data Encryption) 방식** | DBMS에서 데이터를 자동으로 암호화·복호화하는 방식 | 응용프로그램 수정 없이 사용 가능, 관리가 편리 |
| **Hybrid 방식** | API 방식과 Plug-in 또는 TDE 방식을 함께 사용하는 방식 | 보안성과 성능을 함께 고려 가능 |

| 구성요소 | 설명 | 역할 |
|----------|------|------|
| **Model** | 애플리케이션의 데이터와 비즈니스 로직을 담당하는 부분 | 데이터 처리, 데이터베이스 연동 |
| **View** | 사용자에게 화면(UI)을 표시하는 부분 | 화면 출력, 사용자 인터페이스 |
| **Controller** | 사용자의 요청을 받아 Model과 View를 연결하는 부분 | 요청 처리, 흐름 제어 |

# 소프트웨어 아키텍처 패턴 유형

| 아키텍처 패턴 | 설명 | 특징 |
|--------------|------|------|
| **MVC (Model-View-Controller)** | 모델, 뷰, 컨트롤러로 역할을 분리하는 패턴 | 화면과 비즈니스 로직을 분리하여 유지보수성 향상 |
| **Layered Pattern (계층형 패턴)** | 시스템을 여러 계층으로 나누어 구성하는 패턴 | 각 계층이 독립적이며 유지보수가 용이 |
| **Client-Server Pattern (클라이언트-서버 패턴)** | 서비스를 요청하는 클라이언트와 제공하는 서버로 구성하는 패턴 | 중앙 서버에서 자원과 서비스를 관리 |
| **Pipe-Filter Pattern (파이프-필터 패턴)** | 데이터를 여러 처리 단계(Filter)를 거쳐 순차적으로 처리하는 패턴 | 데이터 변환 및 스트림 처리에 적합 |
| **Broker Pattern (브로커 패턴)** | 분산 환경에서 브로커가 클라이언트와 서버를 중개하는 패턴 | 분산 객체 간 통신 지원 |
| **Master-Slave Pattern (마스터-슬레이브 패턴)** | 마스터가 작업을 분배하고 슬레이브가 처리하는 패턴 | 병렬 처리 및 부하 분산에 적합 |
| **Peer-to-Peer (P2P) Pattern** | 모든 노드가 동등한 역할을 수행하는 패턴 | 중앙 서버 없이 자원 공유 가능 |
| **Event-Bus Pattern (이벤트 버스 패턴)** | 이벤트를 통해 컴포넌트 간 통신하는 패턴 | 느슨한 결합, 이벤트 기반 처리 |
| **Blackboard Pattern (블랙보드 패턴)** | 공유 저장소(Blackboard)를 통해 여러 컴포넌트가 협력하는 패턴 | 복잡한 문제 해결에 적합 |
| **Interpreter Pattern (인터프리터 패턴)** | 문법 규칙을 정의하고 해석하여 실행하는 패턴 | SQL, 정규식 등 언어 해석에 사용 |

### 소프트웨어 아키텍처 4+1 View 모델

| **프로세스 뷰 (Process View)** | 시스템의 실행 과정, 프로세스, 스레드, 동시성 및 성능을 표현하는 뷰이다. | 시스템 통합자, 개발자 |
| **구현 뷰 (Implementation View)** | 소프트웨어 모듈, 컴포넌트, 패키지 등 구현 구조를 표현하는 뷰이다. | 개발자 |
| **배포 뷰 (Deployment View)** | 소프트웨어와 하드웨어의 물리적 배치 및 네트워크 구성을 표현하는 뷰이다. | 시스템 엔지니어, 운영자 |

| 기술 | 설명 | 특징 |
|------|------|------|
| **IPSec (IP Security)** | IP 계층에서 데이터를 암호화하고 인증하여 안전하게 전송하는 보안 프로토콜 | VPN 구축에 주로 사용 |
| **SSL/TLS (Secure Sockets Layer / Transport Layer Security)** | 전송 계층에서 데이터를 암호화하여 안전한 통신을 제공하는 프로토콜 | HTTPS의 기반 기술 |

### 객체 지향 설계 원칙

| 원칙 | 설명 | 핵심 키워드 |
|------|------|------------|
| **SRP (Single Responsibility Principle, 단일 책임 원칙)** | 하나의 클래스는 하나의 책임만 가져야 하며, 변경 이유도 하나여야 한다. | 하나의 책임 |
| **OCP (Open-Closed Principle, 개방-폐쇄 원칙)** | 기존 코드를 수정하지 않고 기능을 확장할 수 있어야 한다. | 확장 O, 수정 X |
| **LSP (Liskov Substitution Principle, 리스코프 치환 원칙)** | 자식 클래스는 부모 클래스를 대체해도 정상적으로 동작해야 한다. | 상속, 치환 가능 |
| **ISP (Interface Segregation Principle, 인터페이스 분리 원칙)** | 하나의 큰 인터페이스보다 목적에 맞는 여러 개의 작은 인터페이스로 분리해야 한다. | 인터페이스 분리 |
| **DIP (Dependency Inversion Principle, 의존 역전 원칙)** | 구현 클래스가 아닌 추상화(인터페이스)에 의존해야 한다. | 추상화 의존 |

### TCP 기법

| 기법 | 설명 | 특징 |
|------|------|------|
| **3-Way Handshake** | TCP 연결을 설정하기 위한 과정 | SYN → SYN+ACK → ACK |
| **4-Way Handshake** | TCP 연결을 종료하기 위한 과정 | FIN → ACK → FIN → ACK |
| **흐름 제어 (Flow Control)** | 송신 측의 전송 속도를 수신 측의 처리 속도에 맞추는 기법 | 슬라이딩 윈도우 사용 |
| **혼잡 제어 (Congestion Control)** | 네트워크 혼잡을 방지하기 위해 전송량을 조절하는 기법 | Slow Start, Congestion Avoidance 등 |
| **오류 제어 (Error Control)** | 손실되거나 오류가 발생한 데이터를 재전송하는 기법 | ACK, 재전송(Retransmission) |
| **슬라이딩 윈도우 (Sliding Window)** | ACK를 기다리지 않고 여러 개의 패킷을 연속 전송하는 기법 | 전송 효율 향상 |

# 선점형(Preemptive) · 비선점형(Non-Preemptive) 스케줄링

| 유형 | 설명 | 대표 기법 |
|------|------|----------|
| **선점형 (Preemptive)** | 실행 중인 프로세스를 중단시키고 다른 프로세스에 CPU를 할당할 수 있는 방식 | Round Robin(RR), SRT, Multi-Level Queue(MLQ), Multi-Level Feedback Queue(MLFQ) |
| **비선점형 (Non-Preemptive)** | 실행 중인 프로세스가 종료되거나 CPU를 반납할 때까지 다른 프로세스가 CPU를 사용할 수 없는 방식 | FCFS, SJF, Priority Scheduling, HRN |

# 선점형 스케줄링

| 기법 | 설명 |
|------|------|
| **Round Robin(RR)** | 일정한 시간(Time Quantum) 동안 실행한 후 다음 프로세스로 CPU를 넘기는 방식 |
| **SRT (Shortest Remaining Time)** | 남은 실행 시간이 가장 짧은 프로세스에 CPU를 할당하는 방식 |
| **Multi-Level Queue (MLQ)** | 우선순위에 따라 여러 개의 큐를 구성하고 각 큐에 다른 스케줄링을 적용하는 방식 |
| **Multi-Level Feedback Queue (MLFQ)** | 프로세스의 실행 특성에 따라 큐를 이동시키며 우선순위를 동적으로 조정하는 방식 |

---

# 비선점형 스케줄링

| 기법 | 설명 |
|------|------|
| **FCFS (First Come First Served)** | 먼저 도착한 프로세스부터 순서대로 실행하는 방식 |
| **SJF (Shortest Job First)** | 실행 시간이 가장 짧은 프로세스를 먼저 실행하는 방식 |
| **Priority Scheduling** | 우선순위가 높은 프로세스를 먼저 실행하는 방식 |
| **HRN (Highest Response Ratio Next)** | 응답률(Response Ratio)이 가장 높은 프로세스를 먼저 실행하는 방식으로 기아 현상을 완화한다. |

| 공격 기법 | 설명 | 특징 |
|-----------|------|------|
| **DoS (Denial of Service)** | 하나의 시스템에서 대량의 요청을 보내 서비스를 마비시키는 공격 | 단일 공격자 |
| **DDoS (Distributed DoS)** | 여러 대의 감염된 시스템(Bot)을 이용하여 서비스를 마비시키는 공격 | 분산 공격 |
| **SYN Flooding** | TCP 연결 요청(SYN)을 반복적으로 보내 서버 자원을 고갈시키는 공격 | TCP 3-Way Handshake 악용 |
| **Smurf Attack** | ICMP Echo Request를 브로드캐스트하여 대량의 응답을 발생시키는 공격 | ICMP 기반 |
| **Ping of Death** | 비정상적으로 큰 ICMP 패킷을 전송하여 시스템을 마비시키는 공격 | 패킷 크기 조작 |
| **TearDrop** | 패킷의 Fragment Offset 값을 조작하여 시스템 오류를 유발하는 공격 | IP 단편화(Fragmentation) 악용 |
| **Land Attack** | 출발지 IP와 목적지 IP를 동일하게 설정하여 시스템에 혼란을 주는 공격 | 동일한 IP 사용 |
| **UDP Flooding** | 대량의 UDP 패킷을 전송하여 시스템 자원을 소모시키는 공격 | UDP 기반 |
| **IP Spoofing** | 출발지 IP 주소를 신뢰할 수 있는 IP로 위조하여 시스템에 접근하거나 공격하는 기법 | 출발지 IP 주소 위조 |
| **ARP Spoofing** | 위조된 ARP 메시지로 MAC 주소를 변조하여 통신을 가로채는 공격 | ARP 프로토콜 악용 |
| **DNS Spoofing** | DNS 정보를 위조하여 사용자를 가짜 사이트로 유도하는 공격 | DNS 캐시 오염 |
| **Session Hijacking** | 정상 사용자의 세션을 탈취하여 권한을 가로채는 공격 | 세션 ID 탈취 |
| **MITM (Man-In-The-Middle)** | 통신 중간에서 데이터를 가로채거나 변조하는 공격 | 중간자 공격 |

| **코드 난독화(Code Obfuscation)** | 프로그램 코드를 분석하기 어렵게 변환하여 역공학과 코드 변조를 방지하는 기술 |

# 소프트웨어 공학 및 프로젝트 관리 법칙

| 법칙 | 설명 | 핵심 내용 |
|------|------|----------|
| **브룩스의 법칙 (Brooks' Law)** | 일정이 지연된 프로젝트에 인력을 추가하면 프로젝트가 더욱 지연된다는 법칙 | 인력 추가 ≠ 일정 단축 |
| **파킨슨의 법칙 (Parkinson's Law)** | 업무는 주어진 시간을 모두 사용할 때까지 늘어난다는 법칙 | 일정이 길수록 업무 증가 |
| **피터의 법칙 (Peter Principle)** | 조직 구성원은 자신의 무능력 수준까지 승진하게 된다는 법칙 | 승진의 한계 |
| **호프스태터의 법칙 (Hofstadter's Law)** | 일을 예상한 것보다 항상 더 오래 걸리며, 이를 고려해도 더 오래 걸린다는 법칙 | 일정 예측 어려움 |
| **콘웨이의 법칙 (Conway's Law)** | 소프트웨어 구조는 이를 개발하는 조직의 의사소통 구조를 반영한다는 법칙 | 조직 구조 = 시스템 구조 |
| **파레토 법칙 (Pareto Principle, 80:20 법칙)** | 결과의 80%는 원인의 20%에서 발생한다는 법칙 | 핵심 원인 20%가 결과의 80%를 결정 |

### 데이터링크 계층의 암호화 전송 기술

### 백업 유형

| 유형 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **전체 백업 (Full Backup)** | 모든 데이터를 전체 백업 | 복구가 가장 빠르고 간단 | 백업 시간과 저장 공간이 많이 필요 |

| **차등 백업 (Differential Backup)** | 마지막 **전체 백업 이후 변경된 데이터만** 백업 | 복구가 증분 백업보다 빠름 | 시간이 지날수록 백업 용량이 증가 |

| **부하 테스트 (Load Test)** | 예상되는 정상 부하를 가하여 시스템 성능을 측정 |
| **스트레스 테스트 (Stress Test)** | 시스템의 한계를 초과하는 부하를 가하여 안정성을 확인 |

### 교착상태(Deadlock) 발생 조건

| 조건 | 설명 |
|------|------|
| **상호 배제 (Mutual Exclusion)** | 한 번에 하나의 프로세스만 자원을 사용할 수 있는 상태 |
| **점유와 대기 (Hold and Wait)** | 자원을 점유한 상태에서 다른 자원을 추가로 요청하며 대기하는 상태 |
| **비선점 (No Preemption)** | 다른 프로세스가 사용 중인 자원을 강제로 빼앗을 수 없는 상태 |
| **환형 대기 (Circular Wait)** | 여러 프로세스가 서로가 점유한 자원을 기다리며 원형으로 대기하는 상태 |

### 상호배제(Mutual Exclusion)와 세마포어(Semaphore)

| 구분 | 설명 |
|------|------|
| **상호배제 (Mutual Exclusion)** | 여러 프로세스가 공유 자원에 동시에 접근하지 못하도록 하는 원리 |
| **세마포어 (Semaphore)** | **프로세스 간 상호 배제와 동기화를 보장하는 기법**으로, 공유 자원에 대한 접근을 제어한다. |

### 세마포어 특징

| 항목 | 설명 |
|------|------|
| **목적** | 프로세스 간 **상호 배제** 및 **동기화** 보장 |
| **동작** | `P(Wait)` : 자원 요청, `V(Signal)` : 자원 반납 |
| **이진 세마포어(Binary Semaphore)** | 값이 0 또는 1만 가능하며 하나의 프로세스만 접근 가능 |
| **계수 세마포어(Counting Semaphore)** | 여러 개의 동일한 자원의 개수를 관리 |

### 라우팅 프로토콜

| 구분 | 프로토콜 | 설명 | 특징 |
|------|----------|------|------|
| **IGP (Interior Gateway Protocol)** | **RIP (Routing Information Protocol)** | 동일한 AS(자율 시스템) 내부에서 사용하는 거리 벡터 라우팅 프로토콜 | 거리 벡터(Distance Vector), 홉 수(Hop Count) 사용, 최대 15홉, 30초마다 갱신, Bellman-Ford 알고리즘 |
| | **OSPF (Open Shortest Path First)** | 동일한 AS 내부에서 사용하는 링크 상태 라우팅 프로토콜 | 링크 상태(Link State), Dijkstra 알고리즘, 빠른 수렴, 대규모 네트워크에 적합 |
| | **IS-IS (Intermediate System to Intermediate System)** | 동일한 AS 내부에서 사용하는 링크 상태 라우팅 프로토콜 | 링크 상태 방식, 대규모 ISP 환경에서 사용, 확장성 우수 |
| | **EIGRP (Enhanced Interior Gateway Routing Protocol)** | Cisco에서 개발한 고급 내부 라우팅 프로토콜 | 거리 벡터와 링크 상태의 장점 결합(Hybrid), 빠른 수렴, Cisco 장비 중심 |
| | **EGP (Exterior Gateway Protocol)** | 서로 다른 AS(Autonomous System) 간의 라우팅을 수행하기 위한 외부 게이트웨이 프로토콜이다. | AS 간 라우팅에 사용되는 프로토콜의 분류이며, 현재는 BGP가 사실상 표준으로 사용된다. |
| | **BGP (Border Gateway Protocol)** | 서로 다른 AS 간 라우팅 정보를 교환하는 대표적인 외부 게이트웨이 프로토콜이다. | 경로 벡터(Path Vector) 방식, 인터넷 백본에서 사용, 대규모 네트워크에 적합하다. |

### 알고리즘 및 방식

| 프로토콜 | 방식 | 알고리즘 |
|----------|------|----------|
| **RIP** | 거리 벡터(Distance Vector) | Bellman-Ford |
| **OSPF** | 링크 상태(Link State) | Dijkstra(Shortest Path First) |
| **IS-IS** | 링크 상태(Link State) | Dijkstra |
| **EIGRP** | 하이브리드(Hybrid) | DUAL |
| **BGP** | 경로 벡터(Path Vector) | Path Vector |

### RIP(Routing Information Protocol)

| 항목 | 내용 |
|------|------|
| **정의** | 거리 벡터(Distance Vector) 기반의 내부 게이트웨이 프로토콜(IGP) |
| **메트릭(Metric)** | 홉 수(Hop Count) 사용, OSPF와 BGP등의 라우팅 프로토콜이 최적의 경로를 선택하는 기준이 되는 최적 경로 선택 기준값 |
| **최대 홉 수** | 15홉(16홉은 도달 불가) |
| **라우팅 갱신** | 약 30초마다 전체 라우팅 테이블 갱신 |
| **알고리즘** | Bellman-Ford 알고리즘 사용 |
| **특징** | 구현이 간단하지만 대규모 네트워크에는 부적합 |

|------|------|------|
| **전용 회선 (Leased Line)** | 송·수신자 간 전용 회선을 임대하여 사용하는 방식 | 안정성과 보안성이 높지만 비용이 비쌈 |
| **회선 교환망 (Circuit Switching)** | 통신 전에 전용 회선을 설정한 후 데이터를 전송 | 실시간 통신에 적합, 회선 효율이 낮음 |
| **패킷 교환망 (Packet Switching)** | 데이터를 패킷으로 분할하여 전송 | 회선 이용 효율이 높고 인터넷에서 사용 |
| **공중 교환 전화망 (PSTN)** | 일반 전화망을 이용한 통신 방식 | 구축이 쉽지만 속도가 느림 |
| **ISDN (Integrated Services Digital Network)** | 음성·영상·데이터를 디지털 방식으로 통합 전송 | PSTN보다 빠른 디지털 통신 |
| **X.25** | 패킷 교환 기반의 가상회선 방식 WAN 기술 | 오류 제어 기능 우수, 속도가 느림 |
| **Frame Relay** | X.25를 개선한 고속 패킷 교환 방식 | 오류 제어를 단순화하여 전송 속도 향상 |
| **ATM (Asynchronous Transfer Mode)** | 고정 길이 셀(Cell, 53Byte)을 이용한 고속 전송 방식 | 음성·영상·데이터를 통합 전송 |

| **통합 보안 관리** | **ESM (Enterprise Security Management)** | 여러 보안 장비의 로그와 이벤트를 통합 관리하는 시스템이다. | 중앙 집중식 관리 |
| **통합 위협 관리** | **UTM (Unified Threat Management)** | 방화벽, IDS, IPS, VPN 등 다양한 보안 기능을 하나의 장비에서 제공한다. | 통합 보안 장비 |
| **망간 자료전송** | **CDS (Cross Domain Solution)** | 서로 다른 보안 등급의 네트워크 간 안전한 데이터 전송을 지원하는 보안 솔루션이다. | 망 연계, 자료 전송 |
| **데이터 유출 방지** | **DLP (Data Loss Prevention)** | 중요 정보의 외부 유출을 탐지하고 차단하는 솔루션이다. | 개인정보, 기밀정보 보호 |
| **웹 방화벽** | **WAF (Web Application Firewall)** | SQL Injection, XSS 등 웹 공격을 탐지하고 차단하는 보안 솔루션이다. | 웹 서버 보호 |
| **보안 정보 및 이벤트 관리** | **SIEM (Security Information and Event Management)** | 다양한 보안 장비의 로그를 수집·분석하여 보안 이벤트를 실시간으로 관리하는 시스템이다. | 로그 분석, 이상 징후 탐지 |
| **디지털 저작권 관리** | **DRM (Digital Rights Management)** | 디지털 콘텐츠의 저작권 보호 및 사용 권한을 관리하는 기술이다. | 콘텐츠 암호화, 라이선스 관리 |

### SSL/TLS

### SSL/TLS (Secure Sockets Layer / Transport Layer Security)

| 구분 | 내용 |
|------|------|
| **정의** | 인터넷에서 클라이언트와 서버 간의 통신을 암호화하여 데이터의 기밀성, 무결성, 인증을 제공하는 보안 프로토콜이다. 현재는 SSL을 개선한 TLS가 주로 사용된다. |
| **기능** | **기밀성(Confidentiality)**, **무결성(Integrity)**, **인증(Authentication)**을 제공한다. |
| **동작 방식** | 공개키 암호화로 세션 키를 교환한 후, 대칭키 암호화를 이용하여 데이터를 안전하게 전송한다. |
| **사용 계층** | 전송 계층(Transport Layer)과 응용 계층(Application Layer) 사이에서 동작한다. |
| **대표 활용** | HTTPS, FTPS, SMTPS, IMAPS, POP3S 등 보안 통신에 사용된다. |

### SSL/TLS의 기능

| 기능 | 설명 |
|------|------|
| **기밀성 (Confidentiality)** | 암호화를 통해 제3자가 데이터를 열람하지 못하도록 보호한다. |
| **메시지 무결성 (Integrity)** | 데이터가 전송 중에 위·변조되지 않았음을 보장한다. |
| **상호 인증 (Authentication)** | 디지털 인증서를 이용하여 통신 상대방의 신원을 확인한다. |

### OLAP 연산

| 연산 | 설명 | 특징 |
|------|------|------|
| **Roll-up (롤업)** | 세부 데이터를 상위 개념으로 집계하여 요약하는 연산이다. | 데이터 요약, 차원 축소 |
| **Drill-down (드릴다운)** | 요약된 데이터를 하위 개념으로 세분화하여 상세하게 조회하는 연산이다. | 데이터 상세 조회, 차원 확장 |

| **Pivoting (피벗)** | 행과 열의 축을 회전하여 다양한 관점으로 데이터를 분석하는 연산이다. | 행·열 전환 |
| **Slicing (슬라이스)** | 하나의 차원에서 특정 값을 선택하여 데이터를 조회하는 연산이다. | 특정 조건 선택 |
| **Dicing (다이스)** | 여러 차원에서 여러 조건을 선택하여 부분 데이터를 조회하는 연산이다. | 다차원 조건 선택 |

### 푸트남 모형 (Putnam Model)

| 구분 | 내용 |
|------|------|
| **정의** | 소프트웨어 개발에 필요한 인력과 개발 기간의 관계를 수학적으로 예측하는 비용 산정 모형이다. |
| **기반 이론** | Rayleigh-Norden(레이리-노든) 곡선 |
| **특징** | 개발 인력은 프로젝트 초기에 적고 중간에 증가한 후, 완료 단계에서 다시 감소하는 형태를 따른다. |
| **장점** | 개발 기간과 투입 인력을 예측하여 프로젝트 계획 수립에 활용할 수 있다. |
| **단점** | 대규모 프로젝트에 적합하며, 초기 개발 규모를 정확하게 산정해야 한다. |

### 신기술

| 기술 | 설명 | 특징 |
|------|------|------|
| **디지털 트윈 (Digital Twin)** | 현실 세계의 사물이나 시스템을 가상 공간에 동일하게 구현하여 시뮬레이션 및 분석하는 기술이다. | 현실과 가상을 실시간으로 연동하여 예측 및 최적화 |
| **CPS (Cyber Physical System)** | 물리 시스템과 사이버 시스템을 실시간으로 융합하여 제어하는 시스템이다. | 센서와 네트워크를 이용한 실시간 모니터링 및 제어 |
| **IoT (Internet of Things)** | 사물에 센서와 통신 기능을 탑재하여 인터넷으로 연결하는 기술이다. | 사물 간 정보 수집 및 교환 |
| **M2M (Machine to Machine)** | 사람의 개입 없이 기계와 기계가 직접 통신하는 기술이다. | 기계 간 자동 통신 |
| **클라우드 컴퓨팅 (Cloud Computing)** | 인터넷을 통해 서버, 저장공간, 소프트웨어 등을 제공하는 기술이다. | 자원 공유, 확장성, 비용 절감 |
| **엣지 컴퓨팅 (Edge Computing)** | 데이터를 클라우드가 아닌 단말기나 네트워크 가장자리에서 처리하는 기술이다. | 지연시간 감소, 실시간 처리 |
| **Fog Computing** | 클라우드와 단말기 사이에서 데이터를 처리하는 분산 컴퓨팅 기술이다. | 엣지와 클라우드의 중간 계층 |
| **메타버스 (Metaverse)** | 현실과 가상이 융합된 3차원 가상 세계를 제공하는 기술이다. | 아바타, 가상경제, 몰입형 서비스 |
| **블록체인 (Blockchain)** | 거래 정보를 분산 저장하여 위·변조를 방지하는 분산 원장 기술이다. | 탈중앙화, 무결성 보장 |
| **NFT (Non-Fungible Token)** | 블록체인 기반의 대체 불가능한 디지털 자산을 나타내는 토큰이다. | 고유성, 소유권 증명 |

### 블랙박스 테스트/화이트박스 테스트

# 블랙박스 테스트 (Black Box Test)

| 구분 | 내용 |
|------|------|
| **정의** | 프로그램의 내부 구조나 로직을 고려하지 않고 요구사항과 기능을 기반으로 수행하는 테스트이다. |
| **특징** | 기능 중심 테스트, 사용자 관점에서 수행 |

# 블랙박스 테스트 유형

| 유형 | 설명 |
|------|------|
| **동등 분할 검사 (Equivalence Partitioning)** | 입력 데이터를 유효한 그룹과 무효한 그룹으로 나누어 테스트하는 기법이다. |
| **경계값 분석 (Boundary Value Analysis)** | 입력값의 최소값, 최대값 등 경계 부분에서 오류를 확인하는 기법이다. |
| **결정 테이블 검사 (Decision Table Testing)** | 조건과 동작의 조합을 의사결정 테이블로 작성하여 테스트하는 기법이다. |
| **상태 전이 검사 (State Transition Testing)** | 시스템의 상태 변화와 상태 간 전이를 확인하는 테스트 기법이다. |
| **유스케이스 검사 (Use Case Testing)** | 사용자의 시나리오(유스케이스)를 기반으로 테스트하는 기법이다. |
| **분류 트리 검사 (Classification Tree Method)** | 입력 조건을 트리 구조로 분류하여 테스트 케이스를 도출하는 기법이다. |
| **페어와이즈 테스트 (Pairwise Testing)** | 입력값의 모든 조합 대신 두 개씩의 조합을 고려하여 테스트하는 기법이다. |
| **원인-결과 그래프 검사 (Cause-Effect Graphing)** | 입력 조건과 출력 결과의 관계를 그래프로 표현하여 테스트 케이스를 설계하는 기법이다. |
| **비교 검사 (Comparison Testing)** | 동일한 기능을 수행하는 여러 프로그램의 결과를 비교하여 오류를 찾는 기법이다. |
| **오류 예측 검사 (Error Guessing)** | 경험과 직관을 바탕으로 오류가 발생할 가능성이 높은 부분을 테스트하는 기법이다. |

# 화이트박스 테스트

| 항목 | 내용 |
|------|------|
| **정의** | 프로그램의 내부 구조와 로직을 기반으로 수행하는 테스트이다. |
| **특징** | 코드 중심 테스트, 개발자 관점에서 수행 |
| **주요 기법** | 기초 경로 검사, 제어 구조 검사(조건 검사, 루프 검사, 데이터 흐름 검사) |

# 화이트박스 테스트 유형

| 유형 | 설명 |
|------|------|
| **기초 경로 검사 (Basis Path Testing)** | 수행 가능한 모든 독립 실행 경로를 테스트하는 기법이다. |
| **제어 구조 검사 (Control Structure Testing)** | 프로그램의 제어 구조를 대상으로 수행하는 테스트 기법이다. |
| **조건 검사 (Condition Testing)** | 프로그램의 조건식의 참(True)과 거짓(False)을 테스트하는 기법이다. |
| **결정 검사 (Decision Testing)** | 각 결정문(Decision)의 참(True)과 거짓(False)이 최소 한 번 이상 수행되도록 테스트하는 기법이다. |
| **조건/결정 커버리지 (Condition/Decision Coverage)** | 조건과 결정을 모두 만족하도록 테스트하는 커버리지 기법이다. |
| **변경조건/결정 커버리지 (Modified Condition/Decision Coverage, MC/DC)** | 각 조건이 다른 조건과 독립적으로 결정 결과에 영향을 미치는지 확인하는 테스트 기법이다. |
| **다중조건 커버리지 (Multiple Condition Coverage)** | 조건식의 가능한 모든 조건 조합을 테스트하는 기법이다. |
| **루프 검사 (Loop Testing)** | 반복문의 오류를 확인하는 테스트 기법이다. |
| **데이터 흐름 테스트 (Data Flow Testing)** | 변수의 정의(Define)와 사용(Use) 흐름을 분석하여 테스트하는 기법이다. |

### 죽음의 핑 (Ping of Death)

| 구분 | 내용 |
|------|------|
| **정의** | 정상 크기(65,535Byte)를 초과하는 ICMP 패킷을 여러 조각(Fragment)으로 분할하여 전송한 후, 재조합 과정에서 시스템 오류나 서비스 장애를 유발하는 서비스 거부(DoS) 공격이다. |
| **특징** | 비정상적으로 큰 ICMP 패킷을 분할하여 전송하고, 재조합 과정의 오류를 악용한다. |
| **공격 목적** | 시스템 다운 및 서비스 거부(DoS) |

### 요구사항 분석에 사용되는 기법

| 기법 | 설명 |
|------|------|
| **자료 흐름도(DFD, Data Flow Diagram)** | 시스템에서 데이터가 입력, 처리, 저장, 출력되는 흐름을 그림으로 표현하는 기법이다. |

| **요구사항 명세서(SRS, Software Requirements Specification)** | 소프트웨어 개발 프로세스의 시작인 소프트웨어의 요구사항을 분석하고 정의하는 단계에서 작성되는 최종 산출물 |
