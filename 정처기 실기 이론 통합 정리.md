### 애자일 유형
# 1. XP (Extreme Programming)

| 구분 | 내용 |
|------|------|
| 개념 | 고객 요구사항 변화에 유연하게 대응하기 위해 짧은 개발 주기를 반복하는 애자일 개발 방법론 |
| 목적 | 고객 만족, 빠른 피드백, 소프트웨어 품질 향상 |
| 핵심 가치 | **Communication(의사소통), Simplicity(단순성), Courage(용기), Respect(존중), Feedback(피드백)** |

### XP 주요 실천 기법

| 기법 | 내용 |
|------|------|
| Pair Programming | 두 명의 개발자가 하나의 PC에서 함께 개발하여 코드 품질 향상 |
| TDD(Test-Driven Development) | 테스트 코드를 먼저 작성한 후 실제 코드를 구현 |
| Refactoring | 기능 변경 없이 내부 구조를 개선하여 유지보수성 향상 |
| Continuous Integration(CI) | 개발한 코드를 지속적으로 통합하고 자동 빌드 및 테스트 수행 |
| Small Release | 작은 단위로 자주 배포하여 고객 피드백을 빠르게 반영 |
| Collective Ownership | 모든 개발자가 프로젝트의 모든 코드를 수정 가능 |

---

# 2. Scrum

| 구분 | 내용 |
|------|------|
| 개념 | 반복적·점진적으로 개발하는 애자일 프레임워크 |
| 개발 단위 | Sprint(보통 1~4주) |
| 특징 | 반복 개발, 점진적 개선, 팀 협업, 고객 피드백 반영 |

### Scrum 주요 역할(Role)

| 역할 | 내용 |
|------|------|
| Product Owner | 요구사항 관리, Product Backlog 우선순위 결정 |
| Scrum Master | Scrum 프로세스 관리, 장애 요소 제거 |
| Development Team | 실제 개발 수행 및 Sprint 목표 달성 |

### Scrum 주요 이벤트(Event)

| 이벤트 | 내용 |
|---------|------|
| Sprint Planning | Sprint에서 수행할 작업 계획 수립 |
| Daily Scrum | 매일 약 15분 동안 진행하는 진행 상황 공유 회의 |
| Sprint Review | Sprint 결과물을 고객에게 시연하고 피드백 수집 |
| Sprint Retrospective | Sprint를 회고하고 개선 사항 도출 |

### Scrum 주요 산출물(Artifact)

| 산출물 | 내용 |
|---------|------|
| Product Backlog | 전체 요구사항 목록 |
| Sprint Backlog | Sprint 동안 수행할 작업 목록 |
| Increment | Sprint 종료 후 완성된 실행 가능한 제품 |

| 특징 | 사람 중심, 협업, 의사소통, 빈번한 배포, 지속적인 개선 |

## 설계·아키텍처·UML·패턴

| 분류 | 패턴 | 핵심 설명 |
| --- | --- | --- |
| 생성 패턴 | Singleton | 클래스의 인스턴스를 하나만 생성하고 어디서든 동일한 객체를 사용하게 하는 패턴이다. |
| 생성 패턴 | Factory Method | 객체 생성을 서브클래스에 위임하여 객체를 생성하는 패턴이다. |
| 생성 패턴 | Abstract Factory | 관련 객체들을 하나의 팩토리에서 생성하는 패턴이다. |
| 생성 패턴 | Builder | 복잡한 객체의 생성 과정을 단계별로 구성하여 동일한 생성 절차로 서로 다른 표현 결과를 만들 수 있게 하는 패턴이다. |
| 생성 패턴 | Prototype | 기존 객체를 복제하여 새로운 객체를 생성하는 패턴이다. |
| 구조 패턴 | Adapter | 호환되지 않는 인터페이스를 변환하여 함께 사용할 수 있게 하는 패턴이다. |
| 구조 패턴 | Bridge | 기능 계층과 구현 계층을 분리하여 독립적으로 확장할 수 있게 하는 패턴이다. |
| 구조 패턴 | Composite | 객체들을 트리 구조로 구성하여 부분과 전체를 동일하게 다루는 패턴이다. |
| 구조 패턴 | Decorator | 객체에 동적으로 기능을 추가하는 패턴이다. |
| 구조 패턴 | Facade | 복잡한 서브시스템에 단순한 통합 인터페이스를 제공하는 패턴이다. |
| 구조 패턴 | Flyweight | 공유 가능한 객체를 재사용하여 메모리 사용량을 줄이는 패턴이다. |
| 구조 패턴 | Proxy | 실제 객체에 대한 대리 객체를 두어 접근을 제어하는 패턴이다. |
| 행위 패턴 | Observer | 한 객체의 상태가 변경되면 의존 객체들에게 자동으로 변경 내용을 전달하는 일대다 의존성 패턴이다. |
| 행위 패턴 | Strategy | 알고리즘을 캡슐화하여 상황에 따라 교체할 수 있게 하는 패턴이다. |
| 행위 패턴 | State | 객체의 상태에 따라 동작이 변경되도록 하는 패턴이다. |
| 행위 패턴 | Command | 요청을 객체로 캡슐화하여 실행·취소 등을 처리하는 패턴이다. |
| 행위 패턴 | Template Method | 처리 순서는 상위 클래스에서 정의하고 세부 구현은 하위 클래스에서 수행하는 패턴이다. |
| 행위 패턴 | Mediator | 객체 간 직접 통신을 중재 객체가 담당하여 결합도를 낮추는 패턴이다. |
| 행위 패턴 | Iterator | 컬렉션의 내부 구조를 노출하지 않고 순차적으로 접근하는 패턴이다. |
| 행위 패턴 | Memento | 객체의 이전 상태를 저장했다가 필요할 때 복원하는 패턴이다. |
| 행위 패턴 | Chain of Responsibility | 요청을 처리할 수 있는 객체를 체인으로 연결하여 순차적으로 처리하게 하는 패턴이다. |
| 행위 패턴 | Visitor | 객체 구조와 처리 기능을 분리하여 새로운 연산을 쉽게 추가하는 패턴이다. |
| 행위 패턴 | Interpreter | 문법 규칙을 클래스로 표현하고 문장을 해석하는 패턴이다. |

| 구분 | 핵심 설명 |
| --- | --- |
| EAI | 기업 내 서로 다른 애플리케이션을 연계하여 데이터와 기능을 통합하는 방식이다. |
| Point-to-Point | 애플리케이션을 1:1로 직접 연결하는 방식이다. |
| Hub-and-Spoke | 중앙 허브를 통해 각 애플리케이션을 연결하는 방식이다. |
| Message Bus | 공통 메시지 버스를 통해 애플리케이션 간 데이터를 교환하는 방식이다. |
| Hybrid | 여러 연계 방식을 혼합하여 사용하는 방식이다. |

### 트랜잭션 이상 현상

> 최신 출처: 모의고사 / 정처기 실기 모의고사 22회
> 전체 출처: 모의고사 / 정처기 실기 모의고사 22회, 모의고사 / 정처기 실기 모의고사 19회, 모의고사 / 정처기 실기 모의고사 18회, 모의고사 / 정처기 실기 모의고사 14회, 모의고사 / 정처기 실기 모의고사 10회, 모의고사 / 정처기 실기 모의고사 9회, 미니모의고사 / 미니 모의고사 25회, 미니모의고사 / 미니 모의고사 2회


| 구분 | 핵심 설명 |
| --- | --- |
| Dirty Read | 한 트랜잭션이 아직 커밋되지 않은 다른 트랜잭션의 변경 데이터를 읽는 현상이다. |
| Non-Repeatable Read | 한 트랜잭션 안에서 같은 행을 두 번 읽었을 때, 중간에 다른 트랜잭션의 UPDATE로 인해 결과가 달라지는 현상이다. |
| Phantom Read | 한 트랜잭션 안에서 같은 조건으로 범위 조회를 두 번 수행했을 때, 중간에 다른 트랜잭션의 INSERT 또는 DELETE로 인해 결과 행의 수가 달라지는 현상이다. |
| READ COMMITTED | 커밋된 데이터만 읽도록 하여 Dirty Read는 방지하지만 Non-Repeatable Read나 Phantom Read는 발생할 수 있는 격리 수준이다. |

[예시]

| 단계 | SQL | 결과 |
| --- | --- | --- |
| T1 | `SELECT COUNT(*) FROM orders WHERE status = 'PENDING';` | PENDING 주문 수를 조회한다. |
| T2 | `INSERT INTO orders VALUES (101, 'PENDING');` | 다른 트랜잭션이 조건에 맞는 행을 추가한다. |
| T1 | `SELECT COUNT(*) FROM orders WHERE status = 'PENDING';` | 같은 조건의 재조회 결과가 달라져 Phantom Read가 발생한다. |

### 테스트 커버리지(Test Coverage) 유형

> 최신 기출: 기출 / 2023년 2회
> 전체 출처: 기출 / 2023년 2회


| 구분 | 내용 |
| --- | --- |
| 구문(문장) 커버리지(Statement Coverage) | 프로그램의 모든 실행문(문장)을 한 번 이상 수행하는 기준 |
| 결정(분기) 커버리지(Decision/Branch Coverage) | 모든 분기의 참(True)과 거짓(False)을 한 번 이상 수행하는 기준 |
| 조건 커버리지(Condition Coverage) | 각 조건식의 참(True)과 거짓(False)을 모두 수행하는 기준 |
| 조건/결정 커버리지(Condition/Decision Coverage) | 조건 커버리지와 결정(분기) 커버리지를 모두 만족하는 기준 |
| 변경 조건/결정 커버리지(MC/DC) | 각 조건이 결과에 독립적으로 영향을 미치는지 확인하는 기준 |
| 다중 조건 커버리지(Multiple Condition Coverage) | 조건식의 모든 가능한 조합을 수행하는 기준 |
| 기초 경로 커버리지(Base Path Coverage) | 프로그램의 모든 독립 실행 경로를 한 번 이상 수행하는 기준 |

### 화이트박스 테스트 커버리지 종류

> 최신 기출: 기출 / 2025년 3회
> 전체 출처: 기출 / 2025년 3회


| 구분 | 내용 |
| --- | --- |
| 문장 커버리지 (Statement Coverage) | 모든 실행 문장을 최소 1회 이상 수행하는 테스트<br>가장 기본적인 커버리지<br>키워드 : 모든 문장 실행 |
| 분기 커버리지 (Branch Coverage) | 모든 분기(True/False)를 최소 1회 이상 수행하는 테스트<br>결정문의 모든 분기 확인<br>키워드 : True, False 모두 수행 |
| 조건 커버리지 (Condition Coverage) | 각 조건식의 결과(True/False)가 모두 1회 이상 나오도록 테스트<br>개별 조건만 검사<br>키워드 : 각 조건의 True/False |
| 결정 커버리지 (Decision Coverage) | 각 결정문의 결과(True/False)를 모두 수행하는 테스트<br>분기 커버리지와 동일한 의미로 사용<br>키워드 : 결정 결과 True/False |
| 조건/결정 커버리지 (Condition/Decision Coverage) | 조건 커버리지와 결정 커버리지를 모두 만족하는 테스트<br>조건과 결정을 함께 검사<br>키워드 : 조건 + 결정 |
| 변경 조건/결정 커버리지 (MC/DC) | 하나의 조건만 변경해도 결정 결과가 바뀌는지를 확인<br>각 조건이 결정에 독립적으로 영향을 미치는지 검사<br>항공, 국방 등 안전성이 중요한 시스템에서 사용<br>키워드 : 독립 영향, 안전성 |
| 다중 조건 커버리지 (Multiple Condition Coverage) | 모든 조건의 가능한 조합(True/False)을 모두 테스트<br>가장 강력한 조건 검사 방식<br>예) 조건 2개 → TT, TF, FT, FF<br>키워드 : 모든 조합 |
| 경로 커버리지 (Path Coverage) | 프로그램의 모든 실행 경로를 최소 1회 이상 수행<br>반복문이 많으면 경로 수가 매우 증가<br>가장 높은 수준의 커버리지<br>키워드 : 모든 실행 경로 |

### 교착상태(Deadlock) 발생 조건 / 교착상태 발생 조건

> 최신 출처: 모의고사 / 정처기 실기 모의고사 25회
> 전체 출처: 모의고사 / 정처기 실기 모의고사 25회, 미니모의고사 / 미니 모의고사 22회


| 구분 | 내용 |
| --- | --- |
| 상호 배제(Mutual Exclusion) | 하나의 자원을 동시에 하나의 프로세스만 사용<br>한 자원을 한 번에 하나의 프로세스만 사용할 수 있다. |
| 점유와 대기(Hold and Wait) | 자원을 가진 상태에서 다른 자원을 요청<br>자원을 보유한 채 다른 자원을 추가로 요청하며 기다린다. |
| 비선점(No Preemption) | 할당된 자원을 강제로 회수 불가<br>이미 할당된 자원을 강제로 빼앗을 수 없다. |
| 환형 대기(Circular Wait) | 서로가 가진 자원을 순환적으로 기다림<br>프로세스들이 서로의 자원을 순환적으로 기다린다. |

| 외래키(Foreign Key) | 다른 릴레이션의 기본키를 참조하는 속성이다. |

| 구분 | 순서 | 핵심 설명 |
| --- | --- | --- |
| 3-Way Handshake | SYN → SYN+ACK → ACK | TCP 연결을 수립하기 위해 클라이언트와 서버가 연결 요청과 수신 확인을 교환하는 절차이다. |
